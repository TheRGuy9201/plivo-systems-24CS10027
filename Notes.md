# NOTES
## Design Overview
The goal of this project is to improve the reliability of UDP media streaming over an unreliable network while keeping playback within the required delay budget. Instead of simply forwarding packets, the sender and receiver implement a lightweight Forward Error Correction (FEC) mechanism using XOR parity.

### Step 1 - Custom Wire Protocol
The original sender simply forwarded the packets received from the harness. In this step, a custom packet format was introduced to support future reliability mechanisms. Every outgoing packet now contains:
- 1-byte packet type
- 4-byte sequence number (or group ID for parity packets)
- 160-byte payload
This allows the receiver to distinguish between normal media packets and recovery packets.

### Step 2 - Sender-side XOR FEC
Frames are grouped into pairs (`GROUP_SIZE = 2`). Every frame is transmitted normally as a DATA packet. After both frames of a group are received by the sender, an XOR parity packet is generated by XOR-ing the payloads of both frames. The parity packet is then transmitted immediately.
For every two data packets:
- DATA(Frame 0)
- DATA(Frame 1)
- PARITY(Group 0)
This introduces approximately **50% bandwidth overhead** while allowing recovery from one packet loss per group.

### Step 3 - Receiver-side Packet Processing
The receiver was modified to understand the custom packet format. Instead of immediately forwarding packets to the player, it now:
- Identifies DATA and PARITY packets.
- Stores packets in group-based buffers.
- Maintains which packets of each group have already arrived.
- Stores parity packets separately for future recovery.
Duplicate packets are ignored to avoid unnecessary processing.

### Step 4 - XOR Packet Recovery
Whenever exactly one packet of a group is missing and the corresponding parity packet is available, the receiver reconstructs the missing payload using XOR operations.
Recovered Packet = Parity XOR Remaining Packet
This recovery process happens automatically whenever sufficient information becomes available.

### Step 5 - Packet Forwarding
After storing or reconstructing packets, the receiver forwards every packet that has not already been sent to the player. A forwarding bitmap ensures that duplicate transmissions are avoided.

## Current Limitations
- Packet ordering has not yet been implemented.
- No jitter buffer is present.
- Playback scheduling is not implemented.
- Only one lost packet per group can be recovered.
- If both data packets or the parity packet are lost, recovery is impossible.

## Recommended Grading Delay
The optimal `delay_ms` is still being evaluated through experiments and will be updated after testing Profiles A and B.
